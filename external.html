链接到外部文档

如果你的项目依赖于外部库或工具，以下几个理由使我们不需要为所有的我们项目涉及到的外部源文件生成文档。

节省磁盘空间:
已经有一些现成的文档了，比如网站上可能已经有某种文档了。您可能希望链接到这些现成的页面，而不是在重新生成到本地。
不同的编译速度:
外部项目通常与您自己的项目具有不同的更新频率。让doxygen一次又一次地解析这些外部项目的源代码是没有多大意义的。
内存：
对于非常大的源码树，让doxygen解析所有的源码可能会占用太多的系统内存。通过将源码划分为几个“包”，其中一部分包的源码需要被doxygen解析，而这个包所依赖的所有其他包都是在外部链接的，不要被解析，从而这节省了大量内存。
可用性:
对于一些用doxygen记录的项目，源码可能是不可用的。
版权问题:

如果外部包及其文档是其他人的版权，那么引用它可能比在项目文档中包含它的副本更好，甚至是必要的。当作者禁止副本发布时，这是很有的。如果作者需要用户遵循某些许可条件作为重新发布的先决条件，而您又不想受这些条件的约束，那么最好引用他们的文档副本，而不要包含副本。

如果您的项目符合以上任何一种情况，您可以使用doxygen的标记文件机制。标记文件基本上是标记外部源码的表示。Doxygen可以生成和读取标记文件。

要为项目生成标记文件，只需将标记文件的名称放在配置文件中的GENERATE_TAGFILE 选项之后。

要将一个或多个外部项目的输出与您自己的项目组合起来，您应该在配置文件中的 TAGFILES选项之后指定标记文件的名称。

标记文件通常只包含从doxygen运行位置开始的文档的相对位置。因此，当您在其他项目中包含标记文件时，您必须指定外部文档与此项目相关的位置。通过将(相对)位置分配给在 TAGFILES 配置选项之后指定的标记文件，可以在配置文件中实现这一点。如果使用相对路径，则相对于生成项目HTML输出的目录;因此，从一个项目的HTML输出目录到链接到的另一个项目的HTML输出的相对路径。

例如：
假设您有一个项目proj，它使用两个名为ext1和ext2的外部项目。目录结构如下:
<root>
  +- proj
  |   +- html               HTML output directory for proj
  |   +- src                sources for proj
  |   |- proj.cpp
  +- ext1
  |   +- html               HTML output directory for ext1
  |   |- ext1.tag           tag file for ext1
  +- ext2
  |   +- html               HTML output directory for ext2
  |   |- ext2.tag           tag file for ext2
  |- proj.cfg               doxygen configuration file for proj
  |- ext1.cfg               doxygen configuration file for ext1
  |- ext2.cfg               doxygen configuration file for ext2
配置文件的相关部分如下:
proj.cfg: 
OUTPUT_DIRECTORY  = proj
INPUT             = proj/src
TAGFILES          = ext1/ext1.tag=../../ext1/html \
                    ext2/ext2.tag=../../ext2/html
ext1.cfg: 
OUTPUT_DIRECTORY  = ext1
GENERATE_TAGFILE  = ext1/ext1.tag
ext2.cfg: 
OUTPUT_DIRECTORY  = ext2
GENERATE_TAGFILE  = ext2/ext2.tag
